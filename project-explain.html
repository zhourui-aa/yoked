🔌 接口与实现 — 可插拔设计

每个功能定义为一个接口（service/），具体用哪家厂商由实现类（impl/）决定。 BotApp 只依赖接口，不知道实现细节。换了厂商只需写一个新的 impl，不改主程序。

接口实现类服务商能力
AiService DeepSeekAiServiceImpl DeepSeek API 多轮对话 + Function Calling + 意图识别
ImageGenService SeedreamImageServiceImpl 火山引擎 Ark 文字 → AI 生成图片
VisionService DoubaoVisionServiceImpl 火山引擎 Ark 图片 → AI 描述分析
SpeechService QwenTtsSpeechServiceImpl 阿里云 DashScope 文字 → 语音合成（14 种音色）
（无接口，直接类） WeatherBotService 和风天气 + Photon 城市名 → 实时天气

为什么这样设计？

假设你想把 DeepSeek 换成 OpenAI GPT：

// 只需新建一个类： public class OpenAiServiceImpl implements AiService { // 实现 chat()、chatWithTools() 等 } // BotApp 中改一行： AiService ai = new OpenAiServiceImpl(); // ← 就这一行变了

其余代码（BotApp、ILinkBot、所有 handler）完全不用动。这就是接口的意义。

⚡ Function Calling 详解

这是项目的核心亮点。不再是"关键词匹配 → 调 AI 判断意图 → 执行"的两段式， 而是把 8 个工具一次性注册给 AI，AI 自己决定调用哪个。一次 API 调用完成。

完整流程

1
用户发消息："北京今天天气怎么样？"
BotApp.processTextMessage()
2
本地命令检查：不是"帮助/人设/音色"等精确命令 → 跳过
BotApp.tryHandleLocalCommand()
3
构建工具列表：动态注册所有可用工具（天气/生图/会话/图片追问/文档追问）
BotApp.buildTools(tools, executors, ...)
4
发送 AI 请求：把所有工具定义 + 对话历史一起发给 DeepSeek
ai.chatWithTools(userId, text, tools, executors)
⇢ DeepSeekAiServiceImpl 内部：builder.addFunctionTool(weatherFunc).addFunctionTool(drawFunc)...
5
AI 决策：DeepSeek 分析用户意图，返回 tool_calls: [{name: "get_weather", arguments: {city: "北京"}}]
6
执行工具：从 executors Map 中找到 "get_weather" 的执行函数，传入 {city: "北京"}
executor.apply(args) → WeatherBotService.query("北京") → "北京 晴 25°C ..."
7
第二轮 AI 调用：把工具执行结果发回 AI，AI 组织成自然语言
"北京今天晴天，温度 25°C，体感温度 26°C，适合出行～"
8
发送回复：文字 + 可选语音
bot.sendTextWithTyping() + sendAsVoice()

注册的工具清单

工具名参数触发条件谁来执行
get_weathercity: 城市名始终可用WeatherBotService.query()
generate_imageprompt: 图片描述生图服务可用SeedreamImageServiceImpl.generate()
create_sessionname: 对话名始终可用SessionManager.createSession()
switch_sessionname: 对话名始终可用SessionManager.switchTo()
delete_sessionname: 对话名始终可用SessionManager.deleteSession()
list_sessions（无）始终可用SessionManager.listSessions()
ask_about_imagequestion: 问题有缓存图片VisionService.analyze()
ask_about_documentquestion: 问题有缓存文档返回文档内容给 AI

工具定义在哪里？

BotApp.java → buildTools()
工具定义

用 functionDef(name, description, parameters) 构造 FunctionDefinition，同时把 name → executor lambda 放入 Map。两个列表一起传给 AI。

BotApp.java → functionDef()
快捷构建器

把 name + description + Map<String,?> 参数定义 变成 OpenAI SDK 的 FunctionDefinition 对象。

DeepSeekAiServiceImpl.java → chatWithTools()
FC 执行引擎

接收 tools + executors → builder.addFunctionTool() 注册 → AI 调用 → 检查 tool_calls → 执行 executor → 结果发回 AI → 返回最终文本。

AiService.java → chatWithTools()
接口定义

在接口层声明了统一 FC 方法。任何 AI 服务实现（DeepSeek/OpenAI/Claude）都要实现它。

微信发消息 │ ▼ ILinkBot.startPolling() ← 长轮询线程：getUpdates() 阻塞等待 │ WeixinMessage → toBotMessage() → BotMessage ▼ handlerExecutor ← 单线程保证消息按序处理 │ ▼ BotApp handler lambda ← 消息分类 │ ├─ msg.isVoice() → handleVoice() → 提取文字 → processTextMessage(forceVoice=true) ├─ msg.isImage() → handleImage() → Vision 识图 → 缓存图片 → 发送 ├─ msg.isFile() → handleFile() → 多编码读/PdfBox/POI → AI 总结 → 缓存文档 └─ 文字消息 → processTextMessage() │ ├─ ① tryHandleLocalCommand() ← 帮助/人设/音色/语音模式（零API） ├─ ② wantsVoice 检测 ← 关键词命中即生效 ├─ ③ buildTools() + chatWithTools() ← 统一FC（一次AI调用） └─ ④ chat() 自由对话 ← FC未命中时降级

📁 文件详解

🔴 入口层

bot/BotApp.java
main() 入口

整个程序的大脑。创建所有服务 → 注册消息处理器 → 启动轮询。消息路由全在这里：图片/语音/文件分类处理，文字走 unified FC。还有文件提取（多编码→PDFBox→POI）、TTS 清理、缓存管理等辅助方法。

🟡 SDK 门面层

ilink/ILinkBot.java
SDK 门面

封装 wechat-ilink-sdk 的全部复杂度。提供：登录（二维码 + 过期重试）、startPolling() 长轮询、sendText/Image/File 等发送方法、WeixinMessage→BotMessage 转换。项目其他地方完全不接触 SDK 类型。

🟢 接口层 service/

service/AiService.java
接口

AI 对话核心接口。定义 chat() 多轮对话、chatWithTools() 统一 FC、extractXxxIntent() 意图提取（旧版，逐步废弃）、record() 记录对话、setPersona() 人设管理。

service/ImageGenService.java
接口

生图接口。只有一个方法 byte[] generate(String prompt)。

service/VisionService.java
接口

识图接口。String analyze(byte[] imageBytes, String prompt)。

service/SpeechService.java
接口

TTS 接口。textToSpeech() + 音色管理。

service/WeatherBotService.java
门面类

包装 com.weather 子模块。独立加载配置，提供 query(city) 和 isAvailable()。

🔵 实现层 impl/

impl/DeepSeekAiServiceImpl.java
AiService 实现

项目最核心的实现类。使用 OpenAI Java SDK 调用 DeepSeek API。实现 chat()（带 Session 历史的多轮对话）、chatWithTools()（统一 FC：注册工具→AI判断→执行→回传结果→最终回复）、quickAsk()（单次 JSON 问答）。依赖 SessionManager 管理多会话。

impl/SeedreamImageServiceImpl.java
ImageGenService 实现

调用火山引擎 Ark API 生图。模型：doubao-seedream-5-0-pro。流程：提交 prompt → 获取图片 URL → HTTP 下载 → 返回 byte[]。处理内容安全拦截和超时。

impl/DoubaoVisionServiceImpl.java
VisionService 实现

调用火山引擎 Ark 多模态 API 识图。模型：doubao-seed-2-0-lite。图片 Base64 编码 → 构建 image_url + text 消息 → 返回 AI 描述。

impl/QwenTtsSpeechServiceImpl.java
SpeechService 实现

调用阿里云 DashScope MultiModalConversation API。模型：qwen3-tts-flash。14 种预设音色（Cherry/Serena/Stella/Ethan/Roy 等），HTTP 下载 WAV 音频返回 byte[]。

impl/SessionManager.java
会话管理

管理多用户多会话。数据结构：userId → (sessionName → Session)。支持 CRUD + 语音模式开关 + 人设管理。MAX_HISTORY=200 条，system prompt = persona + TECH_INSTRUCTIONS。

impl/Session.java
数据容器

单个会话：name、persona、roles[]+contents[]（并行列表存对话历史）、add()、trim()。

🟣 模型层 model/

model/BotMessage.java
消息载体

统一消息类型。支持 text/image/voice/file 四种模式。ILinkBot 把 SDK 的 WeixinMessage 转成这个，所有 handler 都用它。不可变对象，静态工厂方法创建。

model/DrawIntent.java
数据类

生图意图结果：isDraw + prompt。旧版两段式意图提取用，FC 重构后逐步废弃。

model/WeatherIntent.java
数据类

天气意图结果：isWeather + city。同上，FC 重构后逐步废弃。

model/ImageFollowUpIntent.java
Record

图片追问判断：boolean isFollowUp。

model/VoiceReplyIntent.java
Record

语音回复意愿：boolean isVoiceReply。

model/SessionCommandIntent.java
Record

会话命令：action(create/switch/delete/list/none) + name。

⚪ 工具层

util/ConfigUtil.java
配置读取

三级优先级：系统属性 -D > 环境变量 > config.properties。所有服务实现类通过它读取 API Key。读取不到返回 null，服务初始化时优雅降级。

com/weather/WeatherApp.java

🔗 关键调用链

用户发"北京天气"的完整链路

微信 App │ 发送 "北京天气" ▼ ILinkBot getUpdates() 长轮询返回 WeixinMessage │ toBotMessage() → BotMessage(text) ▼ BotApp handler msg.isImage()? isVoice()? isFile()? → 都不是 │ 调用 processTextMessage() ▼ tryHandleLocalCommand() "北京天气" 不是"帮助"/"人设"/... → 跳过 ▼ buildTools() 构造 [get_weather, create_session, switch_session, ...] ▼ ai.chatWithTools() │ DeepSeekAiServiceImpl: │ ① 把 8 个 tool 注册到 builder │ ② builder.model(MODEL).build() → API 调用 │ ③ AI 返回: tool_calls = [{function: {name: "get_weather", arguments: {city: "北京"}}}] │ ④ 从 executors Map 取 "get_weather" → executor.apply({city:"北京"}) │ ⑤ WeatherBotService.query("北京") │ → Photon API 地理编码 → 北京 lat=39.9 lon=116.4 │ → 和风天气 API → {temp:25, feelsLike:26, text:"晴", ...} │ → toWeChatFormat() → "🌤 北京 晴 25°C ..." │ ⑥ 构造 followUp 请求，把天气结果作为 tool_result 发回 AI │ ⑦ AI 返回: "北京今天晴天，温度 25°C，适合出行～" ▼ BotApp 收到 reply = "北京今天晴天..." │ bot.sendTextWithTyping(userId, reply, 500L) │ voiceMode? → sendAsVoice() → TTS → 发音频文件

用户发图片的完整链路

微信 App │ 发送一张图片 ▼ ILinkBot WeixinMessage → item.getImage_item() → downloadImageFromMessageItem() │ toBotMessage() → BotMessage.image(userId, bytes, caption) ▼ BotApp handler msg.isImage() == true │ 调用 handleImage(bot, ai, vision, msg) ▼ DoubaoVisionServiceImpl.analyze() │ 图片 Base64 编码 → data:image/jpeg;base64,... │ Ark API 多模态请求: [image_url, text:"请描述这张图片"] │ 返回: "这是一张照片，画面中有一只橘猫..." ▼ BotApp │ LAST_IMAGE.put(userId, CachedImage(bytes)) ← 缓存，5min，支持追问 │ ai.record(userId, "[图片]", result) ← 写入对话历史 │ bot.sendTextWithTyping(userId, result) ← 发送

🧵 线程模型

线程作用关键代码
ilink-poller 长轮询线程。阻塞在 client.getUpdates()，有消息就提交到 handlerExecutor ILinkBot.startPolling()
msg-handler 单线程消息处理器。保证消息按序处理，不会两条消息同时改同一个 Session ILinkBot.handlerExecutor
image-gen 生图专用线程池。生图耗时长(10~30s)，放在后台不阻塞消息处理 BotApp.IMAGE_EXECUTOR
main 主线程。启动后阻塞在 Thread.currentThread().join()，等待 JVM 退出 BotApp.main()
