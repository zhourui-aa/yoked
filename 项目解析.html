🤖 微信 AI 聊天机器人

基于 wechat-ilink-sdk 的多模态 AI Bot — 架构、流程、实现原理全解

一、项目概览

维度内容
语言Java 21 + Maven
微信接入wechat-ilink-sdk v2.3.3（长轮询 + AES-ECB 加解密 + CDN 上传下载）
AI 对话DeepSeek v4-pro（1M token 上下文），通过 OpenAI Java SDK 调用
图片生成火山引擎 Seedream 5.0 Pro（Ark SDK），异步线程执行
图片识别火山引擎 Doubao Seed Vision（多模态 Chat API）

文件解析Apache PDFBox + Apache POI，支持 TXT/PDF/Word/Excel
源文件24 个 Java 文件，约 2200 行代码

二、架构设计

项目采用分层 + 接口隔离架构。每一层只依赖下一层的接口，不依赖具体实现。换 AI / 换识图模型 / 换 TTS 引擎，只需写一个新的实现类，BotApp.java 一行不改。

入口层 BotApp        ← 组装所有组件，启动消息循环
  ├─ 门面层 ILinkBot  ← 封装 SDK 的 Builder、登录、加解密、CDN
  └─ 服务层（全是接口）
       ├─ AiService          → DeepSeekAiServiceImpl
       ├─ VisionService      → DoubaoVisionServiceImpl
       ├─ ImageGenService    → SeedreamImageServiceImpl
       ├─ SpeechService      → QwenTtsSpeechServiceImpl
       └─ WeatherBotService  → WeatherService

为什么用接口？

所有 AI 能力都定义成接口，BotApp 只持有接口引用。比如 AiService ai = new DeepSeekAiServiceImpl()，将来换成 new OpenAiServiceImpl()，BotApp 一行不用改。这就是面向接口编程的核心价值：可插拔、可替换。

三、消息路由 —— 整个 Bot 的心脏

所有消息经过一个统一的优先级路由。语音修饰符提前检测，后续任意回复都可附加音频输出。

收到消息
  ├─ 是语音？ ──→ 提取微信预识别文字 → 走文字路由 → TTS 合成 → 发音频
  ├─ 是图片？ ──→ Vision 识图 → 缓存图片(支持追问) → 记入对话历史
  ├─ 是文件？ ──→ 提取文本(PDF/Word/Excel/TXT) → AI 总结 → 缓存文档
  └─ 是文字？
       ├─ [提前检测] 语音修饰符？ → wantsVoice 标记
       ├─ 会话命令？ → 新建/切换/删除/查看对话、帮助、人设、音色
       ├─ 生图意图？ → 关键词粗筛 → AI 精确判断 → 异步 Seedream
       ├─ 天气意图？ → 关键词粗筛 → AI 精确判断 → 和风天气
       ├─ 图片追问？ → 有缓存图？ → AI 判断是否追问 → Vision
       ├─ 文档追问？ → 有缓存文档？ → 关键词命中 → AI 回答
       └─ AI 自由对话 → wantsVoice 为真则附加 TTS 语音

两段式意图识别（关键词粗筛 + AI 精确判断）

第 1 段：用简单关键词过滤，比如"画、生成、来一张"→ 命中才进入第二步。95% 的聊天消息不命中关键词，零额外 API 消耗。

第 2 段：命中的消息发给 DeepSeek，判断"这真的是生图请求吗？要画什么？"。AI 返回 JSON {"isDraw":true,"prompt":"一只猫"}。

这解决了纯关键词匹配永远写不完规则的问题——"帮我整一个"、"给我搞张"、"能不能来一张"……中文太灵活了，交给 AI 判断。

四、微信 iLink SDK 原理

4.1 登录流程

ILinkClient.executeLogin()
  → GET /api/v2/qrcode → 返回二维码 → 打印到终端
  → 后台线程轮询 GET /api/v2/qrcode/status
     waiting → 继续等
     scanned → 已扫码，等确认
     logged_in → 拿到 botToken + userId + botId → LoginContext
     expired → 抛异常 → Bot 自动重新生成二维码（最多 10 次）

4.2 消息拉取（长轮询）

GET /api/v2/updates  （带 cursor 游标）
  → 服务端阻塞直到有新消息或超时（35 秒 readTimeout）
  → 返回新消息列表 + 新的 cursor
  → SDK 把每条消息的 context_token 存入 ContextPoolManager（按 userId 索引）
  → 下次拉取带新 cursor，实现增量拉取，不会重复接收

4.3 消息发送

发送文字：从 ContextPoolManager 取出该用户的 contextToken → POST /api/v2/message/send。这就是为什么必须先收到用户消息才能回复——contextToken 来自用户发来的第一条消息。

发送图片/文件/语音：先生成随机 AES Key → AES-ECB-PKCS7 加密 → 向微信申请 CDN 上传凭证 URL → POST 加密数据到 CDN → 发送加密引用 + AES Key 给微信 → 微信从 CDN 取密文 → 用 Key 解密 → 转发给用户。

为什么加密？微信 CDN URL 是公开的。不加密的话任何人拿到 URL 都能下载。客户端在上传前用随机 Key 加密，只有通过消息体里的 Key 才能解密。

4.4 SDK 内部架构（63 个类）

ILinkClient（唯一入口，AutoCloseable）
├── LoginService        — 二维码 → 轮询 → LoginContext
├── UpdateService       — 长轮询 + cursorStore 增量拉取
├── MessageService      — 发送文字/图片/语音/文件/视频（含 contextToken）
├── MediaService        — 加密上传 CDN + 下载解密
├── TypingService       — "正在输入..." 状态
├── ContextPoolManager  — 用户 ↔ contextToken 映射
├── HeartbeatService    — 定时心跳 + 自动重连
├── ExecutorManager     — IO 线程池 + 调度线程池
├── RetryPolicy         — 指数退避 + jitter
└── AesEcbUtil          — AES-ECB-PKCS7 加解密

五、图片生成原理

用户: "帮我画一只猫"
  → 关键词"画"命中 → AI 判断 {isDraw:true, prompt:"一只猫"}
  → 异步线程提交到 IMAGE_EXECUTOR（单线程池）
  → POST ark.cn-beijing.volces.com/api/v3
     model: doubao-seedream-5-0-pro-260628
     prompt: "一只猫", size: 1K, format: png, responseFormat: URL
  → 火山引擎返回临时图片 URL
  → HttpClient GET 下载图片 → byte[]
  → ILinkBot.sendImage() → AES 加密 → 上传 CDN → 发送
  → 主线程立即返回继续处理消息（不阻塞）

为什么异步：生图耗时 10~30 秒。如果同步执行会卡住整个消息循环，期间用户发的任何消息都无法被拉取。改为异步后，主线程立即返回继续 poll 新消息。

Seedream 模型原理：文生图扩散模型。prompt → 文本编码成向量 → 初始化纯噪声图 → 迭代去噪（每步参考文本向量引导图像方向）→ 几十步后噪声图变成一只猫。

六、图片识别 + 追问原理

用户发送图片
  → SDK: downloadImageFromMessageItem() → GET CDN → AES 解密 → byte[]
  → Base64 编码 → "data:image/jpeg;base64,/9j/4AAQ..."
  → POST 多模态 Chat API
     content: [{type:"image_url", image_url:{url:"data:..."}},
               {type:"text", text:"请描述这张图片"}]
     model: doubao-seed-2-0-lite-260215
  → 返回文字描述 → 缓存图片 byte[]（5 分钟）
  → 用户追问"照片里有人吗" → 缓存图 + 新问题 → Vision API

为什么用 Base64：HTTP JSON 不能直接传输二进制。Base64 把任意字节映射到 64 个可打印 ASCII 字符。代价：体积膨胀 33%。

为什么缓存图片：微信不支持图文混发——用户发图时不能附带文字 prompt。缓存解决这个问题，让用户可以先发图再问。

七、语音对话原理

7.1 语音输入（零 API 消耗）

微信的 VoiceItem.text 字段自带语音识别结果——微信服务端已做好 ASR。下载语音字节只是为了支持后续的音色克隆等场景，日常对话只用 text 字段。

7.2 TTS 选型过程

方案协议复杂度结论
火山引擎豆包 TTSWebSocket + 自定义二进制帧协议（4字节头、大端序、位操作、状态机）最高❌
阿里云智能语音交互AK+SK→Token→WebSocket高❌
阿里百炼 Qwen-TTS 实时WebSocket + JSON中❌
阿里百炼 qwen3-tts-flashHTTP → 音频 URL → 下载最低✅

7.3 语音气泡失败

尝试了手动解析 WAV 二进制头、Java AudioSystem 提取 PCM、指定 encodeType=4——SDK 不报错但微信端不显示。结论：微信语音气泡只接受 AMR/SILK 编码，Java 生态无可用编码器。降级为文件方式发送。

7.4 语音模式 vs 语音修饰符

语音修饰符：在消息处理开头检测语音意图，存成 wantsVoice 标志。后续天气/AI 对话产生文字回复后，检查标志决定是否附加音频。这样"用语音说一下杭州天气"能同时触发天气查询和语音输出。

语音模式：在 SessionManager 中按用户保存布尔标记。开启后所有回复自动附带音频，无需每次说"发语音"。

八、多会话管理原理

SessionManager
  ├─ Map<userId, Map<sessionName, Session>>  — 每个用户多个会话
  ├─ Map<userId, String> currentSession      — 当前活跃会话名
  └─ Map<userId, Boolean> voiceMode          — 语音模式开关

Session
  ├─ String persona       — 当前人设（可独立修改）
  ├─ List<String> roles   — ["user","assistant","user",...]
  └─ List<String> contents — 对应的消息内容（最多 200 条 = 100 轮）

人设分离：每个会话的 persona 可独立修改，但 TECH_INSTRUCTIONS（"你有语音回复能力，不要说你不能发语音..."）始终追加到所有会话的 system prompt 末尾。这样人设随便玩，语音能力不会"丢失"。

九、文件处理原理

收发文件
  → SDK downloadFileFromMessageItem() → AES 解密 → byte[]
  → 尝试 UTF-8 / GBK / GB2312 读取（检测乱码率 < 5%）
  → 失败？→ 尝试 PDFBox 提取 PDF 文本
  → 失败？→ 尝试 POI 提取 Word 文本
  → 失败？→ 尝试 POI 提取 Excel 文本
  → 都失败 → 提示不支持
  → 成功 → 截断到 8000 字 → AI 总结 → 缓存文档（支持追问）

十、关键技术决策

决策 1：为什么 DeepSeek 用 OpenAI SDK？

DeepSeek API 完全兼容 OpenAI 接口格式。用 OpenAI Java SDK 只需把 baseUrl 指向 api.deepseek.com/v1，不用自己拼 JSON、处理重试。代码从 200 行手动拼接缩减到 20 行。

决策 2：为什么天气用 Photon 而不是 Open-Meteo？

Open-Meteo（GeoNames 后端）对中文城市覆盖率不够。"宿迁"、"周口"等地级市搜不到。Photon 基于 OpenStreetMap，对中国城市的覆盖远好，且免费无需 Key。

决策 3：为什么不支持语音气泡？

微信语音气泡需要 AMR 或 SILK 编码。SDK 的 sendVoice 只负责加密上传，不做转码。TTS 输出 WAV/MP3，和 AMR 不兼容。Java 生态无成熟的 SILK 编码库。产品上接受降级——发音频文件，点开播放。

决策 4：为什么消息收发分离到不同线程？

最初是单线程——pollMessages → 处理 → sendReply → sleep → pollMessages。生图/TTS/API 调用耗时几秒到几十秒，期间完全无法接收新消息。改为轮询线程独立运行后，主线程处理消息的同时轮询线程持续拉取，消息永不丢失。
